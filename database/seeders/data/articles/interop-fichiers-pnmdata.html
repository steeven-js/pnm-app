Fichiers PNMDATA — Anatomie et structure

Les fichiers PNMDATA sont le support de transport des tickets PNM entre opérateurs. Chaque vacation (V10, V14, V19) génère un ou plusieurs fichiers PNMDATA contenant les tickets à envoyer. Comprendre leur structure est essentiel pour le diagnostic des échanges inter-opérateurs.

Convention de nommage

Les fichiers PNMDATA suivent une convention de nommage stricte :

PNMDATA_[OPE]_[OPR]_[DATE]_[VACATION]_[SEQ].dat

  • PNMDATA : préfixe fixe identifiant le type de fichier
  • [OPE] : code opérateur émetteur (02, 15, 20)
  • [OPR] : code opérateur récepteur
  • [DATE] : date de génération au format YYYYMMDD
  • [VACATION] : identifiant de la vacation (V10, V14, V19)
  • [SEQ] : numéro de séquence (en cas de fichiers multiples pour une même vacation)

Exemple : PNMDATA_02_20_20250115_V10_001.dat — Fichier émis par Digicel (02) vers Orange (20), le 15 janvier 2025, vacation de 10h, séquence 1.

Structure du fichier

Chaque fichier PNMDATA est structuré en trois sections :

1. En-tête (Header) :

  • TYPE_FICHIER : "PNMDATA"
  • VERSION : version du protocole (ex : "V3.2")
  • EMETTEUR : code opérateur émetteur
  • DESTINATAIRE : code opérateur destinataire
  • DATE_GENERATION : horodatage de création du fichier
  • VACATION : vacation d'émission
  • NB_TICKETS : nombre de tickets contenus dans le fichier

2. Corps (Tickets) :

Chaque ticket est un enregistrement contenant :

  • NUM_DOSSIER : identifiant unique du dossier de portage
  • TYPE_TICKET : code du ticket (1110, 1210, 1410, etc.)
  • MSISDN : numéro mobile au format international (590690XXXXXX)
  • RIO : Relevé d'Identité Opérateur (présent dans le 1110)
  • CODE_OPR : code opérateur receveur
  • CODE_OPD : code opérateur donneur
  • CODE_OPA : code opérateur attributaire
  • DATE_PORTAGE : date de portage prévue (JP) au format YYYYMMDD
  • MOTIF_REFUS : code motif (uniquement pour les tickets 1210 en refus)
  • HORODATAGE : date et heure de création du ticket

3. Pied de page (Footer) :

  • CHECKSUM : somme de contrôle pour vérifier l'intégrité du fichier
  • FIN_FICHIER : marqueur de fin

Système de vacations

Les fichiers PNMDATA sont échangés selon un calendrier tri-quotidien appelé système de vacations :

  • V10 (vacation 10h) : dépôt avant 10h00, collecte à 10h00. Vacation principale, utilisée pour les bascules du jour JP.
  • V14 (vacation 14h) : dépôt avant 14h00, collecte à 14h00. Vacation intermédiaire pour les réponses et confirmations.
  • V19 (vacation 19h) : dépôt avant 19h00, collecte à 19h00. Dernière vacation de la journée, principalement pour les nouvelles demandes.

Chaque opérateur dépose ses fichiers PNMDATA sur le serveur sFTP du destinataire avant l'heure de vacation. Le destinataire collecte et traite les fichiers à l'heure de vacation.

Validation à réception

À réception d'un fichier PNMDATA, l'opérateur destinataire effectue des contrôles :

  • Vérification du checksum (intégrité du fichier)
  • Vérification de la cohérence de l'en-tête (destinataire correct, vacation valide)
  • Vérification du nombre de tickets déclaré vs. réel

Si le fichier est valide, un ACR (accusé de réception) est généré. Sinon, un ERR (fichier d'erreur) est retourné.
