Cas pratique — Correction d'une incohérence col.3 dans un fichier PNMDATA

Ce guide documente un cas réel rencontré le 04/03/2026 sur le fichier PNMDATA.04.02.20260303123443.002 envoyé par Dauphin Télécom (04) à destination de Digicel (02). Une incohérence a été détectée sur la colonne 3 d'un ticket, nécessitant une correction manuelle avant réintégration.

⚠ Contexte
Ce type d'anomalie survient lorsqu'un opérateur envoie un ticket dont la colonne 3 (OPR — opérateur destinataire) ne correspond pas au destinataire réel du fichier. Le fichier est adressé à Digicel (02), mais un ticket pointe vers Free Caraïbe (06).

1. Détection de l'incohérence

En analysant le fichier avec le Décodeur fichier de PNM App, l'outil détecte automatiquement :

Élément Valeur
Fichier PNMDATA.04.02.20260303123443.002
Ligne concernée Ligne 12
Type de ticket 1430 CP (Confirmation de Portage)
MSISDN 0696861327
Col.3 dans le ticket 06 — Free Caraïbe
Destinataire attendu (entête) 02 — Digicel

La ligne brute incriminée :

1430|04|06|06|03|20260226102713|0696861327|2d404a42c6d2db55116a944148c0327d|0011|20260303123659|20260303000000|

On voit que les colonnes 2 et 3 contiennent 04|06 au lieu de 04|02. Ce ticket aurait dû être dans le fichier PNMDATA.04.06 (Dauphin → Free Caraïbe) et non dans PNMDATA.04.02 (Dauphin → Digicel).

2. Procédure de correction

ℹ Principe
On ne peut pas simplement supprimer la ligne car cela créerait un trou dans la séquence des numéros de ticket. La méthode consiste à remplacer la ligne incohérente par le dernier ticket du fichier, en ajustant le numéro de séquence.

Étape 1 — Identifier les éléments clés

Élément Valeur dans notre cas
Ligne à supprimer Ligne 12 (séquence 0011)
Dernier ticket du fichier Ligne 66 (séquence 0065, ticket 3410 RN pour MSISDN 0690660689)
Nombre total de tickets avant 65 tickets (67 lignes avec entête + pied de page)
Nombre total de tickets après 64 tickets (66 lignes avec entête + pied de page)

Étape 2 — Supprimer la ligne incohérente

Ouvrir le fichier dans un éditeur de texte (Notepad++, VS Code...). Supprimer la ligne 12 :

1430|04|06|06|03|20260226102713|0696861327|2d404a42c6d2db55116a944148c0327d|0011|20260303123659|20260303000000|

Étape 3 — Déplacer le dernier ticket à la place

Prendre le dernier ticket du fichier (ancienne ligne 66) :

3410|04|00|01|04|20260227014913|0690660689|9158f2f3eecf1084d8f0b2b287b877d6|0065|20260227062551|20260303000000

Le placer à l'emplacement de la ligne 12 et changer son numéro de séquence de 0065 à 0011 :

3410|04|00|01|04|20260227014913|0690660689|9158f2f3eecf1084d8f0b2b287b877d6|0011|20260227062551|20260303000000
⚠ Important
Le dernier ticket (ligne 66) est supprimé de sa position d'origine puisqu'il a été déplacé. On passe donc de 65 à 64 tickets.

Étape 4 — Renommer le fichier avec un nouvel horodatage

Le fichier corrigé ne peut pas garder le même nom, sinon il sera rejeté ("fichier déjà reçu"). Il faut générer un nouvel horodatage et un nouveau numéro de séquence :

Élément Avant Après (exemple)
Nom du fichier PNMDATA.04.02.20260303123443.002 PNMDATA.04.02.20260304143000.005
Ligne 1 (entête) 0123456789|PNMDATA.04.02.20260303123443.002|04|20260303123443 0123456789|PNMDATA.04.02.20260304143000.005|04|20260304143000
Dernière ligne (pied de page) 9876543210|04|20260303123443|000067 9876543210|04|20260304143000|000066

Les 3 éléments à mettre à jour :

  1. Nom du fichier : nouvel horodatage + nouveau numéro de séquence
  2. Entête (ligne 1) : même horodatage que le nom
  3. Pied de page (dernière ligne) : même horodatage + compteur de lignes totales (entête + tickets + pied de page = 64 + 2 = 000066)

Étape 5 — Transférer et intégrer

  1. Transférer le fichier corrigé via FileZilla dans le répertoire recv/ sur le serveur vmqproportasync01 : 
    /home/porta_pnmv3/PortaSync/pnmdata/04/recv/
  2. Exécuter le script d'intégration PnmAckManager pour traiter le fichier corrigé
  3. Vérifier les logs : 
    tail -f /home/porta_pnmv3/PortaSync/log/PnmAckManager.log

3. Résumé de la procédure

4. Points de vigilance

  • Ne jamais renuméroter toutes les séquences — c'est trop risqué et long. La méthode du remplacement par le dernier ticket est la plus simple.
  • Toujours renommer le fichier avec un nouvel horodatage, sinon erreur "fichier déjà reçu".
  • Mettre à jour 3 endroits : nom du fichier, entête (ligne 1) et pied de page (dernière ligne).
  • Le compteur en pied de page doit refléter le nombre total de lignes (entête + tickets + pied de page).
  • Vérifier avec le Décodeur fichier de PNM App avant de transférer, pour s'assurer qu'il n'y a plus d'incohérence.
✓ Bonnes pratiques
  • Toujours conserver une copie du fichier original avant modification
  • Documenter l'intervention (date, fichier, ligne corrigée, raison)
  • Informer l'opérateur émetteur (ici Dauphin Télécom) de l'anomalie
