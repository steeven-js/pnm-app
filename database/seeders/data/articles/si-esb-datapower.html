ESB DataPower — Bus d'intégration

L'ESB DataPower (Enterprise Service Bus) est le bus d'intégration central de l'architecture PNM chez Digicel. Basé sur la technologie IBM DataPower Gateway, il assure l'interconnexion entre les différents systèmes impliqués dans le processus de portabilité.

Rôle dans l'architecture PNM

L'ESB DataPower remplit une fonction de médiateur entre les systèmes internes :

  • Routage des messages : acheminer les requêtes du bon émetteur vers le bon destinataire (PortaDB → MOBI, CRM → PortaDB, etc.)
  • Transformation de données : convertir les formats entre systèmes hétérogènes (XML, JSON, formats propriétaires)
  • Orchestration : coordonner les séquences d'appels multi-systèmes nécessaires pour une bascule
  • Sécurité : authentification, chiffrement et contrôle d'accès sur les flux inter-applicatifs

Flux principaux

Les flux PNM transitant par l'ESB DataPower incluent :

Flux PortaDB → MOBI (bascule) :

  • PortaDB envoie un ordre de bascule contenant le MSISDN et les paramètres de routage
  • L'ESB transforme la requête au format attendu par MOBI
  • MOBI exécute les commandes EMA (ajout) et EMM (modification) sur les tables de routage réseau
  • Le résultat (succès ou erreur) remonte vers PortaDB via l'ESB

Flux CRM → PortaDB (demande de portage) :

  • Le CRM soumet une nouvelle demande de portage avec les informations client
  • L'ESB valide le format et transmet à PortaDB pour création du ticket 1110
  • PortaDB retourne un accusé de réception avec le numéro de dossier

Flux PortaDB → CRM (notifications) :

  • PortaDB notifie le CRM des changements de statut (acceptation, refus, bascule effectuée)
  • Le CRM met à jour la fiche client et peut déclencher des notifications SMS au client

Transformations

L'ESB DataPower effectue plusieurs types de transformations sur les messages :

  • Mapping de champs : correspondance entre les noms de champs PortaDB et les noms MOBI
  • Enrichissement : ajout d'informations contextuelles (horodatage, identifiant de corrélation)
  • Validation de schéma : vérification de la conformité des messages avant transmission
  • Gestion des erreurs : capture des exceptions, retry automatique, mise en file d'attente (dead letter queue)

Interfaces DAPI — Endpoints de production

La DAPI (Digicel Application Portability Interface) définit les web services exposés et consommés par les différents composants PNM. Voici les endpoints de production :

WS ESB (consommé par PortaDB) :

  • WSDL : DigicelFwiEsbWs4Porta.wsdl
  • Endpoint : http://172.24.116.76:8000
  • Configuration : l'URL est stockée dans la table web_config de PortaDB avec le code esb.wsdlLocation
  • Usage : appels de PortaDB vers l'ESB pour les ordres de bascule et les notifications

WS DAPI — ESB (exposé par PortaWs) :

  • Service : DigicelFwiPortaWs4Esb
  • Endpoint : http://172.24.119.72:8080/PortaWs/
  • Usage : interface consommée par l'ESB pour interroger ou notifier PortaDB

WS DAPI — PortaSync (exposé par PortaWs) :

  • Service : DigicelFwiPortaWs4PortaSync
  • Endpoint : http://172.24.119.72:8080/PortaWs/
  • Usage : interface consommée par PortaSync pour les opérations de synchronisation et d'échange de données

WS DAPI — WebStore (exposé par PortaWebUi) :

  • Service : DigicelFwiPortaUiWs4Esb
  • Endpoint : http://172.24.119.71:8080/PortaWebUi/
  • Usage : interface consommée par l'ESB pour les opérations via l'interface utilisateur web

Monitoring et logs

Tous les flux transitant par l'ESB sont journalisés. Les logs DataPower permettent de tracer chaque message de bout en bout grâce à un identifiant de corrélation unique. En cas d'incident sur un portage, ces logs sont le premier point d'investigation pour identifier à quelle étape le flux a échoué.
