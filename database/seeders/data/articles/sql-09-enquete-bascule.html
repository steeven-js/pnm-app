Enquête : suivi d'une bascule

Le scénario

Plusieurs alertes peuvent déclencher cette enquête :

  • Email CTO : « Bascule non terminée avant 02h00 »
  • Rapport quotidien : « 2 portages en état bascule sans confirmation 1430 »
  • Demande orale : « Le portage du client est bloqué depuis la bascule »

Rappel du workflow de bascule

Le processus normal :

  1. saisi (ticket 1110) → demande envoyée
  2. encours (ticket 1210) → OPD accepte
  3. bascule (ticket 1410) → bascule technique effectuée
  4. clôturé (ticket 1430) → OPD confirme → FIN

Un portage bloqué en bascule (sans ticket 1430) signifie que la bascule technique est faite, mais l'OPD n'a pas confirmé. Le client est « entre deux opérateurs ».

Étape 1 : Portages en état bascule

SELECT p.id, p.msisdn, p.date_portage, e.label AS etat,
       d.operateur_id_origine AS de, d.operateur_id_destination AS vers,
       p.updated_at
FROM porta_portage p
JOIN porta_etat e ON e.id = p.etat_id_actuel
JOIN porta_dossier d ON d.id = p.dossier_id
WHERE e.classe = 'bascule'
ORDER BY p.date_portage;

Étape 2 : Délai depuis la bascule

Combien de temps le portage est-il bloqué en bascule ?

SELECT
    p.msisdn,
    p.date_portage,
    ph.date_debut AS debut_bascule,
    EXTRACT(DAY FROM (NOW() - ph.date_debut)) AS jours_en_bascule
FROM porta_portage p
JOIN porta_etat e ON e.id = p.etat_id_actuel
JOIN porta_portage_historique ph ON ph.portage_id = p.id
WHERE e.classe = 'bascule'
  AND ph.etat_id_from = 3  -- etat bascule entrante
  AND ph.date_fin IS NULL
ORDER BY jours_en_bascule DESC;

Étape 3 : Vérifier la présence du ticket 1430

Normalement, un ticket 1430 dans un fichier confirme la bascule :

SELECT p.msisdn, p.date_portage,
       d2.code_ticket, d2.id AS data_id
FROM porta_portage p
LEFT JOIN porta_data d2 ON d2.id_portage = p.id_portage
                       AND d2.code_ticket = '1430'
WHERE p.etat_id_actuel = 3  -- bascule
ORDER BY p.date_portage;

Si d2.id est NULL, il n'y a pas de ticket 1430 → bascule non confirmée.

Étape 4 : Portage un jour férié

Un portage dont la date tombe un jour férié ne peut pas être traité :

SELECT p.msisdn, p.date_portage, fd.ferryday
FROM porta_portage p
JOIN porta_ferryday fd ON fd.ferryday = p.date_portage
WHERE fd.is_active = true;

Ces portages doivent être replanifiés au prochain jour ouvré.

Étape 5 : Historique complet d'un portage

Pour un portage spécifique, reconstruire toute la timeline :

SELECT
    ph.date_debut,
    ph.date_fin,
    e.label AS etat,
    e.classe,
    CASE
        WHEN ph.date_fin IS NOT NULL
        THEN EXTRACT(HOUR FROM (ph.date_fin - ph.date_debut)) || 'h '
             || EXTRACT(MINUTE FROM (ph.date_fin - ph.date_debut)) || 'min'
        ELSE 'EN COURS'
    END AS duree
FROM porta_portage_historique ph
JOIN porta_etat e ON e.id = ph.etat_id_from
WHERE ph.portage_id = 42  -- remplacer par l'ID du portage
ORDER BY ph.date_debut;

Vue synthétique des portages problématiques

SELECT
    p.msisdn,
    p.date_portage,
    e.label AS etat,
    op_orig.nom AS de,
    op_dest.nom AS vers,
    EXTRACT(DAY FROM (NOW() - p.updated_at)) AS jours_sans_maj
FROM porta_portage p
JOIN porta_etat e ON e.id = p.etat_id_actuel
JOIN porta_dossier d ON d.id = p.dossier_id
JOIN porta_operateur op_orig ON op_orig.code = d.operateur_id_origine
JOIN porta_operateur op_dest ON op_dest.code = d.operateur_id_destination
WHERE e.is_end = false
  AND p.date_portage < CURRENT_DATE
ORDER BY p.date_portage;

Tous les portages non terminés dont la date de portage est passée — ce sont les cas urgents.

Anomalies à trouver :
  • 2 portages en état « bascule » sans ticket 1430
  • 1 portage avec date_portage = jour férié (Mardi Gras 2026-02-17)
  • 2 portages incomplets (bascule sans clôture depuis plusieurs jours)
