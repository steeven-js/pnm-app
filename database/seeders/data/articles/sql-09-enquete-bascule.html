Enquête : suivi d'une bascule

Le scénario

Plusieurs alertes peuvent déclencher cette enquête :

  • Email CTO : « Bascule non terminée avant 02h00 »
  • Rapport quotidien : « 2 portages en état diffuse sans confirmation (ticket 600) »
  • Demande orale : « Le portage du client est bloqué depuis la bascule »

Rappel du workflow de bascule

Le processus normal :

  1. initial (ticket 100) → demande envoyée
  2. enCours (ticket 200) → OPD accepte
  3. diffuse (ticket 300) → bascule technique effectuée
  4. termine (ticket 600) → OPD confirme → FIN

Un portage bloqué en diffuse (sans ticket 600) signifie que la bascule technique est faite, mais l'OPD n'a pas confirmé. Le client est « entre deux opérateurs ».

Étape 1 : Portages en état diffuse (bascule)

SELECT p.id, p.portage_msisdn, p.portage_date_souhaitee,
       e.etat_name AS etat,
       d.operateur_id_donneur AS de, d.operateur_id_receveur AS vers
FROM porta_portage p
JOIN porta_etat e ON e.etat_name = p.portage_etat
JOIN porta_dossier d ON d.id = p.dossier_id
WHERE e.etat_name IN ('diffuse', 'accepte')
ORDER BY p.portage_date_souhaitee;

Étape 2 : Délai depuis la bascule

Combien de temps le portage est-il bloqué en bascule ?

SELECT
    p.portage_msisdn,
    p.portage_date_souhaitee,
    ph.date AS debut_bascule,
    EXTRACT(DAY FROM (NOW() - ph.date)) AS jours_en_bascule
FROM porta_portage p
JOIN porta_portage_historique ph ON ph.id_portage = p.id
WHERE p.portage_etat IN ('diffuse', 'accepte')
  AND ph.etat = 'diffuse'
ORDER BY jours_en_bascule DESC;

Étape 3 : Vérifier la présence du ticket 600

Normalement, un ticket 600 (confirmation) confirme la bascule :

SELECT p.portage_msisdn, p.portage_date_souhaitee,
       pd.code_ticket, pd.id AS data_id
FROM porta_portage p
LEFT JOIN porta_portage_data pd ON pd.id_portage = p.id
                                AND pd.code_ticket = 600
WHERE p.portage_etat IN ('diffuse', 'accepte')
ORDER BY p.portage_date_souhaitee;

Si pd.id est NULL, il n'y a pas de ticket 600 → bascule non confirmée.

Étape 4 : Portage un jour férié

Un portage dont la date tombe un jour férié ne peut pas être traité :

SELECT p.portage_msisdn, p.portage_date_souhaitee, fd.ferryday
FROM porta_portage p
JOIN porta_ferryday fd ON fd.ferryday = p.portage_date_souhaitee
WHERE fd.is_active = true;

Ces portages doivent être replanifiés au prochain jour ouvré.

Étape 5 : Historique complet d'un portage

Pour un portage spécifique, reconstruire toute la timeline :

SELECT
    ph.date,
    ph.etat,
    ph.commentaire,
    e.etat_code
FROM porta_portage_historique ph
JOIN porta_etat e ON e.etat_name = ph.etat
WHERE ph.id_portage = 42  -- remplacer par l'ID du portage
ORDER BY ph.date;

Vue synthétique des portages problématiques

SELECT
    p.portage_msisdn,
    p.portage_date_souhaitee,
    e.etat_name AS etat,
    op_orig.nom AS de,
    op_dest.nom AS vers
FROM porta_portage p
JOIN porta_etat e ON e.etat_name = p.portage_etat
JOIN porta_dossier d ON d.id = p.dossier_id
JOIN porta_operateur op_orig ON op_orig.code = d.operateur_id_donneur
JOIN porta_operateur op_dest ON op_dest.code = d.operateur_id_receveur
WHERE e.etat_name NOT IN ('termine', 'refuse', 'annule')
  AND p.portage_date_souhaitee < CURRENT_DATE
ORDER BY p.portage_date_souhaitee;

Tous les portages non terminés dont la date de portage est passée — ce sont les cas urgents.

Anomalies à trouver :
  • 2 portages en état « diffuse » sans ticket 600
  • 1 portage avec portage_date_souhaitee = jour férié (Mardi Gras 2026-02-17)
  • 2 portages incomplets (bascule sans clôture depuis plusieurs jours)
