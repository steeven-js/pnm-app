Enquête : refus de portage

Le scénario

Plusieurs scénarios peuvent déclencher cette enquête :

  • Email automatique : « 8 refus de portage (code 500) détectés pour RIO incorrect »
  • Email automatique : « 3 tickets avec code_reponse 20 (erreur format) détectés »
  • Demande orale : « Le client dit que son portage a été refusé, vérifie pourquoi »

Comprendre les codes de réponse

D'abord, consultez les codes existants :

SELECT * FROM porta_code_reponse
WHERE procedure = 'eligibilite'
ORDER BY code;
CodeSignification
3Refus (générique)
10Délai trop court
11Erreur dépassement
20Erreur de format (champ manquant/invalide)
30Erreur de facturation
40Numéro non reconnu

Étape 1 : Trouver tous les refus

SELECT p.portage_msisdn, p.portage_rio,
       pd.code_reponse, cr.description AS motif, pd.date,
       pd.operateur_emetteur, pd.operateur_recepteur
FROM porta_portage_data pd
JOIN porta_code_reponse cr ON cr.code = pd.code_reponse
JOIN porta_portage p ON p.id = pd.id_portage
WHERE pd.code_ticket = 500
ORDER BY pd.date DESC;

Étape 2 : Refus pour numéro non reconnu spécifiquement

SELECT p.portage_msisdn, p.portage_rio,
       cr.description AS motif, pd.date
FROM porta_portage_data pd
JOIN porta_code_reponse cr ON cr.code = pd.code_reponse
JOIN porta_portage p ON p.id = pd.id_portage
WHERE pd.code_ticket = 500
  AND pd.code_reponse IN (3, 40)
ORDER BY pd.date DESC;

Vérifiez la colonne portage_rio : si elle contient des valeurs comme « XX0000000000 », le RIO est clairement invalide.

Étape 3 : Trouver les erreurs de format (code_reponse 20)

Le code_reponse 20 indique une erreur de format dans un ticket :

SELECT pd.id, p.portage_msisdn, p.portage_rio,
       pd.code_ticket, pd.code_reponse, pd.commentaire, pd.date
FROM porta_portage_data pd
JOIN porta_portage p ON p.id = pd.id_portage
WHERE pd.code_reponse = 20
ORDER BY pd.date DESC;

Examinez les colonnes : un portage_rio à NULL ou un id_portage manquant est souvent la cause.

Étape 4 : Statistiques des refus par période

SELECT
    pd.date,
    pd.code_reponse,
    cr.description AS motif,
    COUNT(*) AS total
FROM porta_portage_data pd
JOIN porta_code_reponse cr ON cr.code = pd.code_reponse
WHERE pd.code_ticket = 500
GROUP BY pd.date, pd.code_reponse, cr.description
ORDER BY pd.date DESC, total DESC;

Permet de détecter un pic de refus sur un jour particulier.

Étape 5 : Taux de refus par opérateur

SELECT
    o.nom AS operateur_donneur,
    COUNT(*) FILTER (WHERE pd.code_ticket = 500) AS refus,
    COUNT(*) FILTER (WHERE pd.code_ticket = 200) AS acceptes,
    ROUND(
        COUNT(*) FILTER (WHERE pd.code_ticket = 500) * 100.0 /
        NULLIF(COUNT(*) FILTER (WHERE pd.code_ticket IN (200, 500)), 0),
        1
    ) AS taux_refus_pct
FROM porta_portage_data pd
JOIN porta_operateur o ON o.code = pd.operateur_emetteur
WHERE pd.code_ticket IN (200, 500)
GROUP BY o.nom
ORDER BY taux_refus_pct DESC;

Note avancée : FILTER (WHERE ...) est une syntaxe PostgreSQL qui permet de compter conditionnellement. NULLIF évite la division par zéro.

Anomalies à trouver :
  • 8 tickets avec RIO invalide (code_reponse 3 ou 40) dont certains ont un RIO « XX0000000000 »
  • 3 tickets avec code_reponse = 20 (erreur format)
