Monitoring PNM — Contrôles essentiels

Le monitoring quotidien de la plateforme PNM est indispensable pour garantir la continuité du service de portabilité. Voici les contrôles essentiels à effectuer chaque jour, les indicateurs de santé à surveiller, et les problèmes courants avec leurs résolutions.

Contrôles quotidiens

L'équipe PNM doit effectuer les vérifications suivantes chaque matin avant la vacation V10 :

  • Statut des serveurs : vérifier que vmqproportawebdb01 et vmqproportasync01 sont accessibles et opérationnels (SSH, services actifs)
  • Connectivité sFTP : tester les connexions sFTP vers les serveurs Orange (20) et SFR/Only (15). Un test de connexion simple suffit : sftp -o ConnectTimeout=10 user@host
  • Fichiers de la veille : vérifier que tous les fichiers PNMDATA de la veille ont été envoyés et que les ACR correspondants ont été reçus
  • Files d'attente : contrôler dans PortaDB qu'aucun ticket n'est bloqué en statut intermédiaire depuis plus de 24h
  • Logs d'erreur : consulter les logs de la veille dans /var/log/portadb/ et /var/log/portasync/ pour détecter les erreurs silencieuses
  • Espace disque : vérifier l'espace disponible sur les deux serveurs (seuil d'alerte : 80%)

Indicateurs de santé (KPIs)

Les indicateurs suivants doivent être suivis quotidiennement :

  • Taux de réussite des bascules : pourcentage de bascules exécutées sans erreur. Cible : >98%
  • Taux d'ACR : pourcentage de fichiers PNMDATA envoyés ayant reçu un ACR. Cible : 100%
  • Délai moyen de traitement : temps entre la réception d'un ticket et son traitement. Cible : <30 minutes
  • Nombre de tickets en erreur : dossiers bloqués nécessitant une intervention manuelle. Cible : 0
  • Taux de refus : pourcentage de demandes de portage refusées par nos soins (quand Digicel est OPD). Un taux anormalement élevé peut indiquer un problème de données
  • Divergences PNMSYNC : nombre d'écarts détectés lors de la synchronisation hebdomadaire. Cible : <0.1% du volume total de numéros portés

Problèmes courants et résolutions

Connexion sFTP refusée :

  • Vérifier le certificat SSH (expiration, renouvellement)
  • Contrôler les règles firewall (changement d'IP côté partenaire)
  • Contacter le référent technique de l'opérateur partenaire si le problème persiste

Bascule en erreur (commande MOBI rejetée) :

  • Vérifier le message d'erreur dans les logs DataPower
  • Causes fréquentes : MSISDN déjà présent dans les tables MOBI (doublon), format de commande invalide
  • Relancer manuellement la bascule après correction, ou escalader à l'équipe réseau

Fichier PNMDATA non reçu :

  • Vérifier les logs sFTP (/var/log/portasync/sftp-pull.log)
  • Contrôler si le fichier est bien présent sur le serveur sFTP distant
  • Relancer le script portasync-pull.sh manuellement

Base PortaDB lente ou inaccessible :

  • Vérifier la charge CPU et mémoire de vmqproportawebdb01
  • Contrôler les requêtes longues (locks, deadlocks)
  • Si nécessaire, relancer le service de base de données (en coordination avec l'équipe Infrastructure)

Alerting

Des alertes automatiques sont configurées pour notifier l'équipe en cas d'anomalie : échec de bascule, absence d'ACR après 2 vacations, espace disque critique, ou service PortaSync arrêté. Ces alertes sont envoyées par email et par SMS aux astreintes PNM.
